# 페디클이 이 문제를 해결하려는 이유

* 현재 의료시스템은 환자를 위한 구조가 아니라 보험청구와 행정처리를 위한 구조
* 그 결과 진료과정에서 발생하는 임상정보가 환자와 의사 모두에게 충분히 활용되지 못함
* 페디클은 진료 중 생성되는 임상정보를 구조화해 환자와 의사에게 의미있게 남기는 것을 목표로 함

---

# 문제인식과 페디클의 해결방식

## 1. 의료시스템은 환자와 의사의 진료사고를 담기보다는 청구를 위해 설계되어 있다

### 문제인식

* 기존 EMR은 보험청구와 행정처리를 최우선으로 설계됨
* 환자 임상정보는 최소한의 텍스트로만 남음
* 치료의 흐름, 상태변화, 판단의 맥락은 체계적으로 관리되지 않음
* 결과적으로 기록은 남지만 진료의 의미는 축적되지 않음

### 페디클 해결법

* 청구 중심 기록이 아닌 임상정보 중심 구조를 지향
* 환자 상태와 치료과정을 구조적으로 저장
* 진료 흐름과 판단을 시간축 기준으로 추적관리 가능하도록 설계

### 보완 포인트

* 문제인식은 정확하나 “진료사고”의 의미가 추상적임
* 기존 EMR과의 관계가 암묵적임
* 왜 청구 중심 구조가 임상정보 활용을 막는지 연결이 약함

### 보완한 해결

* 보험청구 중심 EMR을 대체하지 않고 임상사고를 보완하는 구조로 정의
* 진료 중 발생하는 관찰·판단·조치 흐름을 별도 임상정보 레이어로 저장
* 환자 상태와 치료과정을 시간축 기준으로 구조화해 누적
* 행정 기록과 임상사고 기록의 역할을 명확히 분리

---

## 2. 진료 기록은 남지만 의사가 왜 그런 판단을 했는지는 남지 않는다

### 문제인식

* 정형외과 진료에서 중요한 것은 어디가 아픈지가 아니라 왜 그렇게 판단했는지임
* 기존 차팅은 자유 텍스트 위주로 기록됨
* 과거 판단의 근거를 다시 파악하기 어려움
* 다음 내원 시 같은 문제를 처음부터 다시 정리해야 함

### 페디클 해결법

* 진료 기록을 문장이 아닌 문제 단위(Problem Unit)로 저장
* 각 문제를 관찰사실, 판단(의심), 조치로 구분
* 판단의 흐름이 시간에 따라 누적되도록 설계
* 나중에 봐도 왜 이런 판단을 했는지 바로 이해 가능

### 보완 포인트

* Problem Unit 개념이 명확히 정의되지 않음
* 판단이 어떻게 변경·누적되는지 설명이 부족함

### 보완한 해결

* Problem Unit을 “하나의 임상 문제에 대한 사고 단위”로 명시
* 각 문제를 관찰사실–판단(의심)–조치로 구조화
* 동일 문제는 내원 간 이어지는 객체로 관리
* 판단의 추가·변경·폐기 이력이 자연스럽게 누적되도록 설계

---

## 3. 통증과 증상은 맥락이 중요한데 기존 입력 방식은 이를 담기 어렵다

### 문제인식

* 정형외과 통증은 부위, 동작, 상황 같은 맥락이 함께 있어야 의미가 있음
* 자유 입력은 번거로워 실제로는 대충 쓰거나 생략됨
* 결과적으로 임상정보가 제대로 남지 않음

### 페디클 해결법

* 통증과 증상을 하나의 카드 단위로 관리
* 처음에는 부위, 좌우, 통증 정도 같은 핵심 정보만 입력
* 필요할 때만 유발동작, 악화·완화 조건을 확장 입력
* 진료 흐름을 끊지 않으면서 정보가 자연스럽게 쌓이도록 설계

### 보완 포인트

* 왜 자유 입력이 실패하는지에 대한 원인이 다소 간략함
* 구조화 강도의 기준이 드러나지 않음

### 보완한 해결

* 통증·증상을 카드 단위로 관리해 입력 부담 최소화
* 최소 핵심 정보만으로 기록 시작 가능하도록 설계
* 추가 맥락 정보는 필요 시 선택적으로 확장
* 구조화를 강제하지 않고 진료 흐름을 우선하도록 구성

---

## 4. 재진 환자의 변화는 중요한데 기존 시스템에서는 한눈에 보이지 않는다

### 문제인식

* 정형외과 진료의 핵심은 이전 내원 대비 상태 변화임
* 기존 차팅 시스템에서는 이전 기록을 일일이 읽어야 함
* 무엇이 달라졌는지 직관적으로 파악하기 어려움
* 판단이 기억이나 감에 의존하게 됨

### 페디클 해결법

* 같은 문제를 매 내원마다 이어서 업데이트
* 새로 작성하는 대신 기존 문제 카드에 변화된 내용만 추가
* 이번 내원에서 달라진 점을 바로 확인 가능

### 보완 포인트

* “변화”가 무엇인지의 기준이 암묵적임
* 재진 중심 설계라는 메시지가 더 강조되면 좋음

### 보완한 해결

* 동일 문제를 내원 간 이어서 관리하는 구조로 설계
* 기존 문제 카드에 변화된 내용만 추가 기록
* 이전 내원 대비 변화 포인트를 우선적으로 노출
* 재진 진료에 최적화된 비교 중심 UX 제공

---

## 5. 원인과 진단은 대부분 가설인데 기록에는 그 불확실성이 남지 않는다

### 문제인식

* 정형외과에서 원인이나 진단은 대부분 가설 상태임
* 기존 시스템에서는 확정 진단처럼 입력되거나 단순 메모로 남음
* 판단이 바뀌어도 왜 바뀌었는지 추적하기 어려움

### 페디클 해결법

* 원인과 진단을 의심(가설) 상태로 기록
* 여러 개의 의심을 동시에 관리 가능
* 확정되지 않은 상태 그대로 유지
* 판단 변화가 자연스럽게 기록되고 이어지도록 설계

### 보완 포인트

* 가설 상태의 활용 목적이 명확히 드러나지 않음
* 확정 전 단계의 가치가 강조되면 좋음

### 보완한 해결

* 원인·진단을 가설 상태로 명시적으로 구분해 기록
* 복수의 가설을 동시에 관리 가능하도록 설계
* 가설의 유지·변경·소멸 이력을 시간순으로 누적
* 진단 확정 이전의 사고 과정을 임상자산으로 축적

---

## 6. 임상정보가 구조화되지 않아 다시 활용할 수 없다

### 문제인식

* 차팅이 텍스트 중심으로 이루어짐
* 특정 증상과 부위에 매핑된 질병을 추출하기 어려움
* 유사환자 그룹화와 과거사례 기반 추론이 불가능함

### 페디클 해결법

* 임상정보를 구조화해 저장
* 유사환자 그루핑 가능
* 유사사례 제공 기반 마련
* 향후 AI 활용에 유리한 데이터 구조 확보

### 보완 포인트

* 구조화의 목적이 활용 관점에서 더 구체화될 필요 있음
* 단순 저장이 아닌 “재사용”과의 연결이 약함

### 보완한 해결

* 임상정보를 증상·문제·판단 단위로 구조화해 저장
* 유사환자 그루핑과 사례 비교가 가능하도록 설계
* 과거 사례 기반 참고와 추론의 기반 제공
* 향후 AI 분석과 추천에 활용 가능한 데이터 구조 확보

---

## 7. ICD코드는 행정에는 맞지만 실제 진료 언어와는 괴리가 있다

### 문제인식

* ICD코드는 너무 포괄적인 경우가 많음
* 실제 임상에서 사용하는 질병 개념과 맞지 않는 경우 존재
* 적절한 ICD코드를 찾기 어려운 상황 발생

### 페디클 해결법

* 증상과 부위를 사전에 ICD코드와 매핑
* 증상 선택만으로 관련 ICD코드 자동 추려짐
* ICD코드와 실제 질병명 간 사전 매칭 DB 구축
* 행정코드와 임상언어 간의 간극을 완화

### 보완 포인트

* ICD 사용의 불편 지점이 사용자 관점에서 더 드러나면 좋음
* ICD가 왜 임상사고를 왜곡하는지 설명이 보완 가능함

### 보완한 해결

* 증상·부위 입력을 기준으로 ICD코드를 후방에서 매핑
* 의사는 ICD를 직접 찾지 않아도 되는 구조
* ICD코드와 실제 질병명 간 사전 매칭 DB 구축
* 행정 코드와 임상 언어를 분리하되 연결 가능한 구조로 설계

---

## 8. 환자는 진료가 끝나면 무엇을 치료했는지 충분히 이해하지 못한다

### 문제인식

* 진료시간이 짧아 충분한 설명이 어려움
* 환자는 질병과 치료과정을 단편적으로 이해함
* 진료 후 치료내용을 정확히 기억하지 못함

### 페디클 해결법

* 진료내용을 구조화된 형태로 요약
* 진료요약을 환자에게 전달
* 차팅 과정을 환자와 함께 보며 설명 가능
* 환자가 진료 중과 이후 모두 자신의 상태와 치료과정을 이해하도록 도움

### 보완 포인트

* 환자 이해 부족의 원인이 시간 외에도 구조 문제임을 강조하면 좋음
* 진료 중과 진료 후 경험의 차이가 명확하지 않음

### 보완한 해결

* 진료 내용을 구조화해 환자용 요약으로 제공
* 차팅 과정을 환자와 함께 보며 설명 가능하도록 지원
* 환자가 진료 중 이해하고 진료 후에도 다시 확인 가능
* 치료 과정과 이유가 환자에게 명확히 남도록 설계

---

## 9. 기존 의료 시스템은 기록을 남기기 위한 도구이지 생각을 돕는 도구는 아니다

### 문제인식

* 기존 EMR은 법적 기록에는 충실함
* 의사의 실제 판단과 사고를 돕지는 못함
* 의사는 기록은 시스템에 남기고 판단은 머릿속에서만 수행함

### 페디클 해결법

* EMR을 대체하지 않는 보조 시스템으로 설계
* 의사의 사고와 판단을 정리해 주는 도구를 지향
* 기록을 늘리는 것이 아니라 판단을 잘 남기는 것을 목표로 함

### 보완 포인트

* 보조 도구로서의 역할이 더 명확히 정의되면 좋음
* “생각을 돕는다”는 표현이 추상적임

### 보완한 해결

* EMR을 대체하지 않는 임상사고 보조 도구로 명확히 정의
* 의사의 관찰·판단·결정을 정리해 주는 구조 제공
* 기록량을 늘리는 것이 아니라 판단의 질을 남기는 것을 목표로 설정
* 의사의 사고 과정을 시스템이 자연스럽게 정리해 주도록 설계
