
진료 차트(Clinical Chart) 페이지의 UX/UI 디자인 컨셉을 정의해줘.

[페이지 맥락]
이 페이지는 의사가 진료 중 환자의 상태를 판단하면서
통증·증상·원인·경과·치료 정보를
S-T-D(Situation / Treatment / Duration) 기반의
구조화된 임상 데이터로 기록하는 핵심 화면이다.

이 화면의 목적은
‘완벽한 의료 기록 작성’이 아니라
‘판단에 필요한 최소 정보를 빠르고 안전하게 남기는 것’이다.

---

[디자인 목표]

1. 진료 흐름을 방해하지 않을 것
2. 입력 피로를 최소화할 것
3. 구조화된 데이터가 자연스럽게 쌓이도록 유도할 것
4. 기록이 임시(Draft)이며 언제든 수정 가능하다는 심리적 안전감을 줄 것

---

[디자인 전제 조건 – 반드시 지킬 것]

- 의사는 입력자가 아니라 판단자다
- 한 화면 = 한 판단 단위
- 모든 입력은 자동 Draft 저장 전제
- 통증/증상/원인/S-T-D 같은 내부 개념을
  UI 탭이나 섹션명으로 직접 노출하지 말 것
- 필수 입력은 최소화하고, 확장은 선택 사항으로 제공할 것

---

[권장 UX 접근 방식]

### 1. 질문 중심 흐름 (Question-driven UX)

폼을 먼저 보여주지 말고,
의사가 실제로 속으로 하는 질문 순서로 화면을 구성할 것.

예시 흐름:

- 지금 가장 불편한 증상은?
- 어디가 불편한가?
- 언제부터 시작되었는가?
- 원인으로 의심되는 맥락이 있는가? (선택)

---

### 2. Progressive Disclosure

- 기본 상태에서는 최소 입력만 노출
  (주증상 1개, 위치 1개, 기간 1개)
- “자세히 기록하기” 같은 명시적 트리거로만
  추가 입력(강도, 양상, 악화/완화 요인 등)을 노출

---

### 3. 구조화 + 자유 서술 혼합

- 모든 선택형 입력에는 보조 텍스트 입력을 허용할 것
- 정답을 요구하지 말고 ‘의심’, ‘추정’ 뉘앙스를 유지할 것

---

[시간·경과(Duration) 표현 가이드]

- Duration은 탭이나 별도 페이지로 분리하지 말 것
- 범주형 선택(오늘/며칠/몇 주)을 우선 제공
- 필요 시 타임라인 형태로 확장 가능하되,
  한 화면에서 과도한 스크롤을 유발하지 말 것

---

[저장 & 안전성 UX]

- 사용자는 “저장”을 의식하지 않아도 되도록 설계할 것
- 입력 중단/이탈/재진입 시 데이터가 유지된다는 신뢰를 줄 것
- 오류/실패 시 사용자 책임처럼 보이지 않도록 설계할 것

---

[절대 하지 말 것 – Anti-Patterns]

- 통증/증상/원인을 탭으로 분리
- 필수 입력 필드 과다
- 단 하나의 저장 버튼
- 팝업 안에 팝업
- 입력 도중 데이터 소실 가능성
- “필수입니다”, “선택하세요” 같은 강압적 문구

---

[디자이너를 위한 판단 질문]

디자인 과정에서 항상 스스로에게 질문할 것:

1. 이 화면은 입력을 위한가, 판단을 위한가?
2. 이 정보는 지금 꼭 필요한가?
3. 나중에 입력해도 되는가?
4. 실수했을 때 복구 가능한가?
5. 이 UI가 진료 흐름을 끊을 가능성은 없는가?

---

[산출물 요구]

- 와이어프레임 또는 UI 시안
- 단, ‘왜 이렇게 구성했는지’를
  위 원칙과 연결해 설명할 수 있어야 함
- 픽셀 단위 디테일보다
  정보 구조와 흐름이 우선이다
