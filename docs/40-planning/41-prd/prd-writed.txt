우리 프로젝트의 전체 MVP 범위를 정의하는 마스터 PRD를 작성해줘.

[프로젝트 맥락]
이 프로젝트는 환자 의료기록을 자유 텍스트가 아닌
S-T-D(Situation / Treatment / Duration) 핵심 축 기반의
구조화된 임상 데이터로 저장하여,
향후 검색·분석·유사 환자 추천·임상 추론에 활용 가능한
데이터 자산을 만드는 것이 목적이다.

[작업 목표]
의사가 환자를 진료(Encounter)하는 전 과정을 기준으로,
임상 데이터의 입력 → 저장 → 변경/추적 → 활용 가능성까지를
단일 PRD 문서로 일관되게 정의할 것.

[포함될 주요 기능 범위]

- 테넌트 & 사용자 전제
  - 의료기관(테넌트) 단위 사용
  - 모든 데이터는 tenant_id 기준 완전 격리

- 임상 기록(차팅) 엔진
  - 환자 선택 및 Encounter 생성
  - Situation / Treatment / Duration 축 기반 구조화 입력
  - 자유 텍스트는 보조 수단으로만 사용

- 기록 라이프사이클
  - 임상 데이터 생성 / 수정 / 정정 흐름 정의
  - 모든 변경은 추적 가능(Audit/History) 전제
  - 수정·삭제 허용 여부는 PRD에서 단정하지 말 것

- 지식 연결 및 활용
  - ICD-10/11, 해부학적 구조(Anatomy)와의 연결 가능성 고려
  - 모든 데이터는 “향후 유사 환자 추천 Feature가 될 수 있는가” 관점에서 정의

[작성 규칙]

- PRD는 prd-writer의 출력 템플릿 순서를 반드시 따른다:
  1) 프로젝트 핵심
  2) 사용자 여정(Clinical Flow)
  3) 제품 핵심 규약(정책 확정분만)
  4) 기능 명세
  5) 메뉴 구조
  6) 페이지별 상세 기능
  7) 데이터 모델(요구 수준)
  8) Policy Decision Request

- 모든 기능은 F001부터 시작하는 기능 ID(Fxxx)를 부여한다.
- [기능 명세] ↔ [메뉴 구조] ↔ [페이지별 상세 기능] 간 정합성을 절대 깨지 말 것.
- 모든 기능/데이터/이벤트는 tenant_id 격리를 기본 전제로 기술한다.
- 의료적 판단, 합법성, 허용/금지/조건부 여부가 필요한 지점은
  PRD에서 결론 내리지 말고 반드시 Policy Decision Request(P-REQ)로 분리한다.
- API URL, 구현 코드, 인프라 상세는 작성하지 않는다.

- 파일이 이미 존재할 경우 전체 내용을 덮어쓴다.
- Markdown 형식으로 작성한다.
